# skills-tinky

A personal library of [Claude Code](https://claude.com/claude-code) skills — categorized, symlink-installed, and self-maintaining. Start a session, tell it what you want, and let it route you to the right tools.

---

## The idea

Each skill is a folder with a `SKILL.md`. They're organized into categories for humans, installed as **flat symlinks** for Claude Code (which discovers personal skills only one level deep), and kept healthy by a small meta-layer:

- **`/god`** — the front door. Describe a goal and it recommends the right skill(s) and hands you a plan to reach it.
- **`/concierge`** — keeps the repo clean and strong (audits naming, frontmatter, structure, symlinks).
- **`skill-creator`** — authors new skills, placing them in the correct category automatically.

---

## Install

```bash
git clone git@github.com:tinkybeatz/skills-tinky.git
cd skills-tinky
./link-skills.sh        # symlinks every skill into ~/.claude/skills/
```

Then **restart Claude Code** (or start a new session) so it picks up the skills.

**How it works:** Claude Code discovers personal skills only at `~/.claude/skills/<skill>/SKILL.md` (one level deep). `link-skills.sh` walks this repo at any depth, links every skill folder by name into `~/.claude/skills/`, aborts on name collisions, and prunes stale links. So the repo's category folders are decoupled from the flat install layout.

**Updating:** `git pull` alone updates existing skills live (they're symlinks). Re-run `./link-skills.sh` only when skills are **added, removed, or moved**.

> Skills point at this repo, so editing a skill here updates it live. Project-specific behavior belongs in each project's `CLAUDE.md`, not in the skill.

---

## The meta-layer

| Skill | Invoke | Role |
|---|---|---|
| **god** | `/god i want to ...` | Understands your objective (asks a couple of questions if vague), recommends the smallest set of skills that reach it, and returns a **sequenced plan** — which skill, in what order, how each helps, and how their outputs chain. It plans and hands off; it doesn't do the downstream work itself. |
| **concierge** | `/concierge` | Repo maintenance. Runs a read-only health check (SKILL.md casing, frontmatter, kebab-case names, duplicate names, taxonomy placement, subfolder-naming, symlink health) and proposes fixes. Warns, never blocks. |
| **skill-creator** | `create a skill` | Authors new skills and agents (with eval loops + benchmarking). For this repo it confirms the target category, enforces naming/uniqueness, scaffolds in the right folder, and offers to link + audit. |

## Categories

The taxonomy is function-first (see [`CATEGORIES.md`](CATEGORIES.md) for the full definitions and each category's `_category.md` conventions).

| Category | Purpose |
|---|---|
| `meta/` | Manage Claude Code, this repo, or the skill toolchain |
| `dev/` | Software engineering: architecture, code, tests, review, ADRs |
| `infra/` | Ops, deploy, databases, sysadmin, security, static analysis |
| `design/build/` | Create interfaces, design-to-code, per-project design setup |
| `design/refine/` | Improve an existing interface along one dimension |
| `product/` | Product strategy, prioritization, business modeling, review |
| `docs/` | Generate formal document deliverables (specs, standards, reports) |
| `notes/` | Notion workspace workflows (requires Notion MCP) |
| `research/` | Research, explanation, and media/content ingestion |
| `personal/` | Personal-life & side-project domains, organized into subfolders |

## Maintenance

- **Add a skill:** use `skill-creator` — it places the skill in the right category and offers to link it.
- **Check repo health:** run `/concierge` (or `bash meta/concierge/scripts/audit.sh`).
- **Re-link after adding/moving skills:** `./link-skills.sh`
- **Regenerate this cheat-sheet:** `./generate-catalog.sh`

### Concierge commit gate

Commits are gated on the concierge audit so structural drift can't land silently:

- **Enable the git hook once per clone** (`core.hooksPath` is local, not versioned):
  ```bash
  git config core.hooksPath .githooks
  ```
  After that, `git commit` runs `meta/concierge/scripts/precommit-gate.sh` and is **blocked on any `[WARN]`**. Bypass intentionally with `git commit --no-verify`.
- **Inside Claude Code**, a `PreToolUse` hook (`.claude/settings.json`) blocks Claude's own `git commit` calls the same way, and reminds that the *judgment* half of concierge (trigger coverage, category fit, overlap) isn't script-checkable — run `/concierge` after content or placement changes.

The git hook covers the mechanical checks for every commit path; the Claude hook additionally nudges the judgment review. Neither replaces running `/concierge` when you reshape skills.
