# BRIEF-WRITER — Validation checklist

Check **before every delivery**. If an item fails, fix it before
delivering. Do not deliver a brief that fails a critical item (★).

---

## 1. Structure

- [ ] ★ The recommendation / conclusion is in the first 3 lines
- [ ] ★ One objective per brief (if multiple objectives → separate briefs)
- [ ] ★ The bullets / findings / options are MECE (no overlap, no gaps)
- [ ] The structure template matches the identified audience
- [ ] Each section opens with its conclusion, not its context

## 2. Content

- [ ] ★ No invented information or anything absent from the DOCUMENTOR report
- [ ] ★ The report's nuances are preserved (moderate confidence ≠ certainty)
- [ ] The report's contradictions are handled (simplified or detailed by audience)
- [ ] Assumptions are flagged or omitted (never presented as facts)
- [ ] Recommendations are actionable (who does what, when)
- [ ] Objectives are measurable (no "improve visibility")

## 3. Tone and language

- [ ] ★ No forbidden phrasing (see voice.md — "it should be noted that", "it is worth pointing out", etc.)
- [ ] Active voice ("We recommend", not "It is recommended")
- [ ] No unsourced superlatives ("market leader" with no source)
- [ ] Jargon matched to the audience (plain for a decision-maker, precise for a technical reader)
- [ ] Unavoidable technical terms explained in a parenthesis (non-technical audiences)

## 4. Length

- [ ] ★ Within the limits of the target audience:
  - Decision-maker client: ≤ 1 page (400-600 words)
  - Technical client: ≤ 4 pages
  - Designer/stakeholder brief: ≤ 15 lines
  - Sales proposal: ≤ 2 pages
  - Creative brief: ≤ 1 page
  - Design brief: ≤ 3 pages

## 5. Traceability

- [ ] "Sources" section present in an appendix (except designer brief and creative brief)
- [ ] Date of the research stated
- [ ] Overall confidence level mentioned
- [ ] Link or reference to the source DOCUMENTOR report

## 6. Type-specific

### If a creative brief

- [ ] The GET describes a person, not a demographic segment
- [ ] The BY is an insight (the "why"), not an observation (the "what")
- [ ] The TO is a measurable behavior

### If a sales proposal

- [ ] Sources are woven in as proof of credibility, not as references
- [ ] The next steps include a concrete timeline
- [ ] The context is reworded in the client's language, not in DOCUMENTOR language

### If a designer/stakeholder brief

- [ ] The options are listed (not just the recommendation)
- [ ] The main risk is 1 line max
