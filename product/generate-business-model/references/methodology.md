# Methodology

## Goal

Create the most relevant business model for an existing web or mobile project by combining project evidence, user answers, research, benchmarks, and explicit assumptions.

## 1. Project Intake

Extract:

- Product: what exists, maturity, feature set, platform, technical constraints.
- User: end user, buyer, payer, decision-maker, non-paying beneficiary.
- Market: country, language, sector, competition, regulation.
- Traction: users, usage frequency, retention, revenue, waitlist, sales, pilots.
- Economics: variable costs, infrastructure, AI/API costs, support, store fees, payment fees, CAC.
- Distribution: SEO, app stores, ads, partnerships, outbound, community, marketplace liquidity.

If artifacts exist, inspect them before questioning the user.

## 2. Hypothesis Map

For each project, build four hypothesis groups:

| Risk | Question | Evidence to seek |
|---|---|---|
| Desirability | Do users strongly want this? | Interviews, retention, waitlist, usage frequency, organic sharing |
| Willingness to pay | Who pays and why now? | Preorders, paid pilots, pricing tests, competitor pricing |
| Feasibility | Can the team deliver reliably? | Tech stack, delivery cost, support load, compliance |
| Viability | Can revenue exceed CAC + COGS + support + platform fees? | Unit economics, margin, churn, payment/store costs |
| Defensibility | Why will the model get stronger with scale? | Data loops, network effects, switching costs, brand, integrations |

Use Strategyzer's desirability/feasibility/viability framing and Lean Startup hypothesis testing logic.

## 3. Candidate Generation

Generate 2-4 business model candidates. For each:

- revenue stream
- pricing metric
- free vs paid boundary
- target segment
- acquisition channel
- core KPI
- main risk
- why it fits or does not fit the project

## 4. Evidence Scoring

Score each candidate from 1 to 5:

| Criterion | Meaning |
|---|---|
| User pull | Users already show repeated need or urgency |
| Payment fit | The payer naturally captures economic value |
| Margin fit | Variable costs are covered at expected price |
| Channel fit | Acquisition channel matches LTV and sales motion |
| Market fit | Model is accepted in target country/sector |
| Defensibility | Scale improves the product or economics |
| Operational simplicity | Team can execute without excessive complexity |

Do not choose only the highest theoretical upside. Penalize models that require marketplace liquidity, heavy compliance, high CAC, or long sales cycles if the team cannot support them.

## 5. Recommendation

Recommend:

- one primary model for the next 3-6 months
- one fallback or expansion model
- experiments that would invalidate or confirm the recommendation

State:

- why this model is best now
- what must be true for it to work
- what data would change the recommendation
